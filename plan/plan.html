Goal

Produce four assignment deliverables that solve data integrity, privilege isolation, scale, and secret rotation for a serverless geospatial data lake on AWS.

Architecture

Event-driven ingestion via S3 → EventBridge → Step Functions, with checksum validation, quarantine routing, and a Glue PySpark anonymizer. Encryption-at-rest with a KMS key policy hardened against privileged-admin override using explicit deny + SCP backstop. Skew handled with AQE + two-stage salted aggregation. Weather API secret rotated via the 4-step Lambda contract with live verification before promotion.

▶Lambda source 
import hashlib, json, logging
import boto3
LOG = logging.getLogger(); LOG.setLevel(logging.INFO)
s3 = boto3.client("s3")
CHUNK = 8 * 1024 * 1024

def _stream_md5(bucket, key):
    body = s3.get_object(Bucket=bucket, Key=key)["Body"]
    digest, total = hashlib.md5(), 0
    for c in iter(lambda: body.read(CHUNK), b""):
        digest.update(c); total += len(c)
    return digest.hexdigest(), total

def _load_manifest(bucket, key):
    return json.loads(s3.get_object(Bucket=bucket, Key=key)["Body"].read())

def lambda_handler(event, _ctx):
    bucket, key, mkey = event["bucket"], event["key"], event["manifestKey"]
    expected = _load_manifest(bucket, mkey)["md5"].lower()
    computed, size = _stream_md5(bucket, key)
    return {
        "bucket": bucket, "key": key, "manifestKey": mkey,
        "checksumValid": computed == expected,
        "computedMd5": computed, "expectedMd5": expected,
        "sizeBytes": size,
    }

State machine shape
ValidateChecksum (Lambda · retry · catch→Failure)
   ├─ valid:false → QuarantineParallel
   │                ├─ CopyToQuarantine → DeleteFromLanding   (aws-sdk:s3 direct)
   │                └─ PublishQuarantineAlert                  (sns:publish direct)
   │                → ChecksumFailed (Fail)
   └─ valid:true  → ExtractZip (Lambda)
                    → StartGlueAnonymize  (glue:startJobRun.sync, --gps_precision 3)
                    → GlueResultChoice
                       ├─ SUCCEEDED → IngestionSucceeded
                       └─ *         → FailureNotification → IngestionFailed
Key design decisions
  • glue:startJobRun.sync blocks on Glue completion natively — no polling Lambda.
  • Optimized SDK integrations for S3 and SNS in the quarantine path — zero Lambda hops.
  • ResultPath discipline: each task augments $, never replaces it, so bucket/key survive end-to-end.
  • Retries with exponential backoff on Lambda and Glue tasks; global Catch on States.ALL → FailureNotification.

Task 4 · KMS Key Policy

~2 hrs
Files
  • Create · deliverables/02-kms-key-policy.json
Policy statements (logical)
  1. EnableLimitedRootAccess — root gets describe/list only (AWS forbids fully locking root out).
  2. AllowKeyAdminOnlyByKmsKeyAdminWithMFA — only KmsKeyAdmin role with fresh (<15 min) MFA can edit the key policy.
  3. AllowGlueDecryptViaService — Glue role can Decrypt/GenerateDataKey, gated on kms:ViaService = glue.region.amazonaws.com + matching encryption context.
  4. AllowBreakGlassDecryptWithMFA — BreakGlassDataAccess role with MFA, last 15 minutes.
  5. DenyDecryptToEveryoneElse — explicit Deny kms:Decrypt to all principals not in {Glue, BreakGlass}. Beats AdministratorAccess.
  6. DenyKeyPolicyTampering — explicit Deny on kms:PutKeyPolicy, kms:Schedule/DisableKey, grants — to all but KmsKeyAdmin.
  7. DenyKeyPolicyTamperingWithoutMFA — same, but unconditional on missing MFA.

Defense, in four layers

Task 6 · Glue job skeleton (skew-resilient)

~2 hrs
Files
  • Create · deliverables/03-glue-job-skeleton.py
Three orthogonal fixes
  • Spark AQE with skew-join — splits skewed partitions at runtime (5× median, 256 MB threshold).
  • Two-stage salted aggregation — salt hot key with 64 random buckets, aggregate twice. Converts one hot partition into 64 balanced ones.
  • Output partition by time only — never partitionBy("intersection_id"); pre-shuffle on a fine spatial hash before writing.
▶Salted aggregator (excerpt) 
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionFactor", "5")
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes", "268435456")

SALT = 64

def aggregate_salted(df):
    salted = df.withColumn("_salt", (F.rand(seed=1729) * SALT).cast("int"))
    stage1 = salted.groupBy("intersection_id", "_salt").agg(
        F.count("*").alias("partial_pings"),
        F.avg("lat").alias("partial_lat"),
        F.avg("lon").alias("partial_lon"),
    )
    return stage1.groupBy("intersection_id").agg(
        F.sum("partial_pings").alias("ping_count"),
        F.avg("partial_lat").alias("centroid_lat"),
        F.avg("partial_lon").alias("centroid_lon"),
    )

write_ready = (
    df.withColumn("lat", F.round("lat", 3))
      .withColumn("lon", F.round("lon", 3))
      .withColumn("event_date", F.to_date("event_ts"))
      .withColumn("event_hour", F.hour("event_ts"))
      .repartition(400, "region", "event_date", "event_hour", "intersection_id")
)
write_ready.write.mode("append")\
    .partitionBy("region", "event_date", "event_hour")\
    .parquet(TARGET)

Task 7 · One-page technical note

~2 hrs
Files
  • Create · deliverables/03-data-skew-tech-note.md
Note structure
  1. Symptom — 10M pings on one intersection_id ⇒ one shuffle partition OOMs while the rest sit idle.
  2. Root cause — default HashPartitioner sends a key to a single partition regardless of cluster size. Worst-case partition is unbounded in n.
  3. Fix 1 · AQE with skew-join (config snippet inline).
  4. Fix 2 · Two-stage salted aggregation with code excerpt referencing 03-glue-job-skeleton.py.
  5. Fix 3 · Output partitioning by time + region only; pre-write repartition on a finer spatial hash.
  6. Rejected approaches — broadcast join (wrong query type), bumping spark.sql.shuffle.partitions (still one hot partition), file splitting (data skew not file skew), manual hot-key carve-out (ops burden).
  7. Result — runs on existing G.1X / 10-worker; peak executor memory drops from >12 GB to <2 GB.

Task 8 · Rotation Lambda (verify-before-promote)

~2.5 hrs
Files
  • Create · deliverables/04-rotate-weather-api-key.py
Four-step contract
  1. createSecret — call Weather admin POST /keys with admin bearer (separate secret), stash result as AWSPENDING.
  2. setSecret — no-op; the API tracks the key from creation.
  3. testSecret — call live GET /v1/current with the pending key; abort on non-200.
  4. finishSecret — move AWSCURRENT from old → new only after testSecret passed, then revoke old key on the admin API.
Safety properties
  • AWSCURRENT never moves until verification passes — the old key keeps serving prod traffic.
  • Every step is idempotent (Secrets Manager will retry).
  • Old-key revocation failures don't roll back rotation (the new key already works).

def test_secret(arn, token):
    pending = json.loads(sm.get_secret_value(
        SecretId=arn, VersionId=token, VersionStage="AWSPENDING"
    )["SecretString"])
    resp = http.request("GET", f"{WEATHER_API_URL}/v1/current?lat=0&lon=0",
                        headers={"X-Api-Key": pending["api_key"]})
    if resp.status != 200:
        raise RuntimeError(f"AWSPENDING key did not authenticate: {resp.status}")

def finish_secret(arn, token):
    meta = sm.describe_secret(SecretId=arn)
    current = next((v for v,s in meta["VersionIdsToStages"].items()
                    if "AWSCURRENT" in s), None)
    if current == token: return
    sm.update_secret_version_stage(
        SecretId=arn, VersionStage="AWSCURRENT",
        MoveToVersionId=token, RemoveFromVersionId=current,
    )
