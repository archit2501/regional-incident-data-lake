Regional Incident Data Lake · AWS architecture submission

High-velocity public safety data, secured before it becomes intelligence.

External agencies stream GPS pings, incident logs, and vehicle telemetry into one regional lake. The submission defends two points that matter most — data integrity before analytics, and privilege isolation before anyone reads raw location data.

P1Orchestration

Automated ingestion pipeline

They asked for

A serverless, event-driven workflow for new .zip arrivals in S3 — Step Functions orchestrated MD5 validation against a manifest, quarantine + SNS routing on failure, extraction on success, and a Glue job that rounds GPS to 3 decimal places and writes Parquet.

What I built

An ASL state machine modeled as a deterministic finite automaton. The validation Lambda streams the S3 object in 8 MiB chunks (Merkle–Damgård MD5 — O(n) time, O(1) memory) so multi-GB zips don't blow Lambda's 10 GB RAM ceiling. Failure fans out S3 copy + delete + SNS publish through a Parallel state. Success calls glue:startJobRun.sync.

Why it works

Streaming hashes inherit O(1) memory from the iterative Merkle–Damgård construction. Direct AWS SDK integrations eliminate Lambda hops on auxiliary steps. Full-jitter exponential backoff on retries (Marc Brooker, 2015) prevents synchronized retry storms under upstream throttling — provably better tail latency than equal-jitter or capped exponential.

Pros · why this approach wins

  • O(n) time / O(1) memory integrity hashing — independent of file size, via streaming Merkle–Damgård chunks.
  • Zero Lambda cold start on S3/SNS/Glue calls — Step Functions optimized SDK integrations remove the wrapper Lambda layer.
  • Parallel fan-out on failure — quarantine copy + delete + SNS publish run concurrently; latency = max(branches), not sum.
  • glue:startJobRun.sync — Step Functions blocks natively on Glue completion; no polling Lambda, no busy loop.
  • Full-jitter retries with JitterStrategy: FULL — proven optimal under contention; breaks thundering-herd retry storms.
  • RFC 3986 percent-encoded copySource — handles any UTF-8 S3 key, including spaces, +, #, non-ASCII.
  • O(1) input gate — 32-char lowercase-hex regex on the manifest fails fast before paying for a multi-GB hash.
  • Bounded execution — top-level TimeoutSeconds=7200, Glue task =5400; a stuck job never sits forever.

Advanced techniques · references

  • Merkle–Damgård construction — Damgård 1989, "A Design Principle for Hash Functions" (CRYPTO). The iterative compression that makes MD5 streamable.
  • Full-jitter backoff — Marc Brooker, "Exponential Backoff and Jitter," AWS Architecture Blog 2015. Formal proof that rand(0, cap) is better than equal-jitter under contention.
  • Workflow as Petri net — van der Aalst, "Workflow Patterns," DKE 2003. Step Functions states map to places/transitions; the visual graph is the Petri net.
  • RFC 3986 — URI percent-encoding for the S3 CopySource header.
  • MD5 caveat — Wang & Yu 2005, Stevens 2013 show chosen-prefix collisions. Used here for transmission integrity against a trusted manifest, not adversarial tamper resistance. Production hardening: S3 Object Lock + SHA-256 KMS-signed manifest.
  • AWS Lambda streaming I/O — boto3's StreamingBody exposes a chunked iterator; we use iter(lambda: body.read(CHUNK), b"") to bound memory.
  • ✓MD5 streamed in Lambda, manifest validated as 32-char hex.
  • ✓Failed files copy to quarantine with tags and metadata, then alert SNS.
  • ✓Passed files go to Glue anonymization with GPS precision = 3.

Decisions & rationale

01Step Functions, not EventBridge Pipes / a Lambda chain / Glue Workflow⌄

Why: Step Functions gives native branching (Choice), built-in Retry/Catch with backoff and jitter, direct SDK integrations for S3/SNS/Glue (no Lambda hop), and a visual execution graph that operators can inspect during failures. It's the right tool the instant the workflow needs more than one decision.

Rejected alternatives:

  • EventBridge Pipes — no branching logic; can't route the same input to either quarantine or Glue.
  • Lambda chain — every transition needs custom retry/error handling; no visual debugging; extra cold starts.
  • Glue Workflow — Glue-centric; quarantine + SNS routing doesn't fit cleanly.
  • MWAA / Airflow — infrastructure overhead, no real payoff for a four-step event-driven flow.
02Direct SDK integrations (no Lambda wrappers around S3, SNS, Glue)⌄

Why: aws-sdk:s3:copyObject, sns:publish, and glue:startJobRun.sync let Step Functions call AWS APIs directly. Zero Lambda cold start per call, no Lambda IAM proliferation, smaller error surface, and .sync makes the state machine block on Glue completion natively — removing the polling Lambda entirely.

Rejected: Wrapping each call in a Lambda. It's the obvious junior approach, and each wrapper costs roughly 50–200 ms cold start, plus a Lambda ARN to maintain and audit.

03URL-encode CopySource inside the Lambda, not in the ASL⌄

Why: The validation Lambda runs urllib.parse.quote(key, safe='/') and returns copySource as a typed field. The state machine then consumes $.validation.copySource directly. Keys containing spaces, +, #, or non-ASCII characters now succeed S3 CopyObject.

How we got here: The parallel-agent review caught this. The original ASL used States.Format('{}/{}', $.bucket, $.key) which produces an unencoded CopySource and fails at runtime for any non-trivial agency key. Moving the encoding into the Lambda makes it unit-testable too.

04Strict 32-char hex validation on the manifest md5⌄

Why: A manifest containing "md5": "" or "md5": "TBD" would otherwise let us hash a multi-GB file and compare against garbage. We fail fast at manifest parse time and route to quarantine with failureReason: "manifest_md5_not_hex".

Rejected: Accept any string and let the natural comparison fail. Wastes bandwidth and disk I/O on a file whose manifest is already broken, and obscures the root cause in the alert.

05Full jitter on every retry + per-task TimeoutSeconds⌄

Why: Pure exponential backoff is correct for one execution but synchronises retries when many executions hit a throttled upstream simultaneously. JitterStrategy: "FULL" spreads retries across the backoff window. Top-level TimeoutSeconds: 7200 + Glue-task TimeoutSeconds: 5400 bound the whole execution so a stuck Glue job doesn't sit forever.

How we got here: The first draft used plain backoff. The agent-execution integration pass surfaced both issues and they were folded into every Retry block.

P2Zero Trust

Ghost Admin defense

They asked for

A KMS Key Policy that lets the Glue Service Role decrypt while explicitly denying kms:Decrypt to all other principals, including those with AdministratorAccess, unless they satisfy a specific security condition (MFA / Break-Glass). Plus a written explanation of how to stop an admin from overriding the policy.

What I built

Defense in depth across four orthogonal control planes: (1) resource-level explicit Deny in the key policy; (2) conditional Allow on Glue gated by kms:ViaService + kms:EncryptionContext; (3) Organizations SCP denying kms:PutKeyPolicy account-wide; (4) CloudTrail Lake + EventBridge auto-remediation via AWS Config.

Why it works

IAM evaluation rule: any explicit Deny matching the request denies it regardless of Allow statements — proven formally by AWS Zelkova's SMT-backed policy verifier. SCPs evaluate at the Org level and cannot be overridden from inside the workload account; only a principal in the management account can change them. Encryption context acts as Additional Authenticated Data (AES-GCM AAD), so decryption with the wrong context fails by construction.

Pros · why this approach wins

  • Formally provable — explicit Deny overrides Allow under AWS IAM semantics; verifiable by AWS Zelkova (Z3 SMT solver) for the no-decrypt-without-MFA property.
  • Orthogonal control planes — Lampson's defense-in-depth: four statistically independent failure surfaces; single-control breach does not compromise the system.
  • Trust-boundary separation — SCPs live in the Org management account; workload admins literally cannot reach them, regardless of IAM privilege.
  • Cryptographic binding — kms:EncryptionContext:project=incident-data-lake is bound as AAD; wrong context invalidates the auth tag.
  • Service-path enforcement — kms:ViaService prevents stolen Glue credentials from being replayed via the CLI; the request principal won't carry the via-service indicator.
  • ArnNotLike with both forms — covers both iam::ROLE and sts::assumed-role/X/*; avoids the canonical IAM policy bug that denies the legitimate Glue role at runtime.
  • Recoverability anchored at root — keeping kms:* for root is the AWS-documented recovery pattern; root containment moves to the SCP layer.
  • Verifiable audit trail — CloudTrail Lake + EventBridge + Config provides the "verifiable audit trail" the Zero-Trust requirement demands.

Advanced techniques · references

  • AWS Zelkova — Backes et al., CAV 2018, "Semantic-based Automated Reasoning for AWS Access Policies Using SMT." Encodes IAM policy as logical formulas and solves with Z3.
  • Envelope encryption — KMS uses two-tier keys: a Data Encryption Key (DEK) encrypted by a Key Encryption Key (KEK) inside the HSM. kms:Decrypt only decrypts the DEK; bulk decryption happens locally on the calling side. Documented in NIST SP 800-57 Part 1.
  • AES-GCM AAD binding — NIST SP 800-38D. Encryption context is incorporated into the GCM authentication tag; tampering invalidates the tag deterministically.
  • Defense in depth — Saltzer & Schroeder 1975, "The Protection of Information in Computer Systems"; Lampson 1974, "Protection." Orthogonal controls = independent failure surfaces.
  • STS assumed-role ARNs — AWS IAM Reference: a role's runtime principal is the STS form arn:aws:sts::ACCT:assumed-role/Role/Session, not the IAM form. ArnLike with the wildcard session is the correct match.
  • AWS Well-Architected SEC02-BP01 — "Limit access to root, administrative users." The Org SCP + sealed root credential is the documented pattern.
  • ✓Glue decrypt limited to service-context use, S3/KMS read path supported.
  • ✓Break-glass requires recent MFA and is auditable.
  • ✓SCP enforcement, root recovery, and kill-chain controls documented.

Decisions & rationale

01Explicit Deny in the resource-based key policy, not IAM-side⌄

Why: IAM evaluation rule — explicit Deny anywhere beats any Allow. AdministratorAccess grants kms:* on Resource *, but a Deny in the key policy applies to the resource itself and overrides the IAM Allow. This is the only way to keep AdministratorAccess intact for other resources while denying decrypt on this single key.

Rejected: Removing kms:* from AdministratorAccess — requires maintaining a custom Admin role and breaks routine operations. SCPs alone — work at the Org level but can't enforce per-key conditions like kms:ViaService or kms:EncryptionContext.

02ArnNotLike with BOTH the IAM and STS assumed-role ARN forms⌄

Why: At runtime, an assumed role's principal ARN is arn:aws:sts::ACCOUNT:assumed-role/RoleName/SESSION, not arn:aws:iam::ACCOUNT:role/RoleName. ArnNotEquals against the IAM form alone would mean the legitimate Glue role gets denied at runtime — the explicit Deny evaluates true for Glue itself.

How we got here: This was the highest-impact bug the parallel-agent review caught. Fix: ArnNotLike matching arn:aws:iam::123…:role/X AND arn:aws:sts::123…:assumed-role/X/* for both the Glue and BreakGlass roles. Verified by re-reading the IAM principal-ARN docs and tracing what sts:AssumeRole actually returns.

03Organizations SCP — the real admin-override defense⌄

Why: The key policy alone isn't enough — an admin with kms:PutKeyPolicy could rewrite the policy and remove the Deny. Org SCPs evaluate at the Organizations level; an admin inside the workload account cannot override them. Only a principal in the Org management account can change SCPs, and that account has separate credentials, MFA, and audit trail. That's what makes the protection real.

Rejected:

  • Just the key policy — the obvious answer, but it can be rewritten by anyone with kms:PutKeyPolicy.
  • Permission boundaries on Admin — brittle, easy to mis-scope, can be lifted by the admin themselves.
  • CloudTrail alerts only — detective; fires after the breach has already happened.
04Preserve root kms:* for documented recoverability⌄

Why: A KMS key with no admin path is permanently unrecoverable. AWS's standard pattern keeps kms:* for the root principal as a recovery anchor. Under normal operation root is blocked by the Org SCP (DenyRootPrincipalExceptRecoveryAndBilling); recovery requires lifting the SCP from the management account — which is itself audited end to end.

How we got here: The initial draft restricted root to read-only actions. The review pass flagged the lockout risk — if KmsKeyAdmin were accidentally deleted or its trust policy broke, the key would be lost. Restored the standard pattern and pushed root containment to the SCP layer.

05kms:ViaService + kms:EncryptionContext:project conditions on the Glue allow⌄

Why: kms:ViaService binds the Allow to a specific AWS service path — a stolen Glue role credential cannot be replayed by a human via the CLI because the request principal won't carry the via-service indicator. kms:EncryptionContext:project = incident-data-lake further binds the decrypt to this data lake; even if Glue ran on another project's data, this key won't decrypt it.

Rejected: Allow the Glue role unconditionally — far too easy to misuse a stolen credential. ViaService includes s3.us-east-1.amazonaws.com as well, because SSE-KMS reads on the Glue path go through S3.

P3Scale

Glue data skew optimization

They asked for

A one-page strategy for 10 million GPS pings concentrated on a single intersection — without raising Glue Worker Type or Max Capacity. The deliverable is a one-page technical note covering PySpark optimizations, S3 partitioning, and salting techniques.

What I built

Three orthogonal fixes applied together: (1) Spark AQE skew-join handling + adaptive coalescing; (2) two-stage salted aggregation — salt the hot key with 64 random buckets (F.rand()*64).cast("int"), aggregate by (intersection_id, salt) emitting sum + count, then strip the salt and reconstruct via Σsum / Σcount; (3) time-only output partitioning with a random-salt pre-write repartition.

Why it works

The skew is Zipfian — a few intersections carry most of the traffic, the classical empirical distribution for road incidents. Hash partitioning a Zipfian key has unbounded worst-case partition size. Salting transforms one O(n)-row partition into 64 partitions of O(n/64) rows. The weighted reconstruction is the unbiased statistical estimator; avg(avg) is biased when bucket sizes differ.

Pros · why this approach wins

  • Bounded worst-case partition size — O(n/S) where S=64 salt buckets, instead of unbounded under naive hash partitioning of a Zipfian key.
  • Mathematically unbiased — weighted reconstruction Σsum_b / Σcount_b is the true sample mean; avg(avg) is a biased estimator under unequal bucket sizes (textbook stratified-sampling result).
  • Constraint-compliant — works on the existing G.1X / 10-worker config; no spark.executor.memory override; no worker-type bump.
  • Memory bound preserved — peak executor memory drops from >12 GB to <2 GB; OOM eliminated structurally.
  • Orthogonal to AQE — AQE handles skewed joins; salting handles the groupBy case AQE doesn't cover (until Spark 3.5+).
  • Query-aligned output partitioning — time-based partition pruning is monotonic and selective; intersection_id pruning would be Zipfian-useless.
  • Deterministic with reproducibility — F.rand(seed=1729) gives identical bucket assignment on retry; useful for incremental re-processing.
  • Bounded write fan-out — pre-write repartition on a random salt expression (not the hot key tuple) — the hot intersection spreads across ~400 / (regions × hours) writer tasks instead of one.

Advanced techniques · references

  • SkewTune — Kwon et al., SIGMOD 2012, "SkewTune: Mitigating Skew in MapReduce Applications." Foundational paper on dynamic task splitting for skewed keys; the conceptual ancestor of all modern salting approaches.
  • Spark AQE — Databricks 2020, "Adaptive Query Execution in Spark." The OptimizeSkewedJoin rule splits skewed shuffle partitions at runtime using observed statistics.
  • High Performance Spark — Karau & Warren, O'Reilly 2017, Ch. 6. Salting/key-splitting recipes; the canonical practitioner reference.
  • Zipf's law for incident data — empirically observed in NHTSA crash statistics: incident density follows a Pareto distribution by intersection. A few hotspots account for most events.
  • Stratified sampling estimator — Cochran, "Sampling Techniques" 3e (1977), Ch. 5. The unbiased estimator from sub-samples is Σ(n_i · mean_i) / Σ n_i, not mean(mean_i). The salted-aggregation correctness proof.
  • Spatial indexing alternatives — Uber's H3 (hexagonal hierarchical), Google's S2 (sphere-spatial), and Morton/Z-order curves (Iceberg/Hudi support) for locality-preserving partitioning.
  • Power-of-two-choices — Mitzenmacher 2001. Theoretical basis for why a small number of salt buckets dramatically smooths load distribution.
  • ✓Salting spreads the hot key across 64 buckets instead of one executor.
  • ✓Weighted sum/count math avoids biased avg(avg).
  • ✓The note explicitly rejects bigger workers and memory-only fixes.

Decisions & rationale

01Two-stage salted aggregation — not bigger workers, not more partitions⌄

Why: The hot intersection produces 10 M rows that the default HashPartitioner sends to one partition. Salting the group key with a random suffix splits the load across 64 buckets at the cost of one extra shuffle of just 64 rows per hot key in the second stage. The salted stage parallelises N-way; the strip-salt stage is tiny.

Rejected:

  • Bigger Glue workers — explicitly forbidden by the assignment, and doesn't fix the per-key bottleneck (the hot partition is per-key, not per-cluster).
  • spark.sql.shuffle.partitions = 4000 — the hot key still maps to one partition; the rest just get smaller. Marginal effect.
  • Broadcast join — only helps joins, not groupBy. Doesn't apply.
  • Splitting input files — the skew is in the data, not the layout. All 10 M rows end up in the same partition after the shuffle anyway.
02Random-salt expression in the pre-write repartition (not intersection_id)⌄

Why: A tuple repartition like repartition(400, region, date, hour, intersection_id) looks correct — more keys = more hashes? — but the hash is computed on the tuple. All rows with the same tuple value still go to one partition, so the hot intersection is funneled right back. Replacing intersection_id with (F.rand() * SALT_BUCKETS).cast("int") breaks the hot key across many writer tasks.

How we got here: This was the second bug the review caught after the avg-of-avgs math. Easy to miss because "more keys → less skew" is the intuitive (and wrong) mental model. The fix is one line in 03-glue-job-skeleton.py.

03sum + count math, not avg of avg⌄

Why: Under random salting, bucket sizes vary by chance. avg(avg(x)) is not avg(x) when group sizes differ — the larger buckets get under-weighted. Emitting sum(x) and count from stage 1 and computing Σsum / Σcount in stage 2 reconstructs the true average exactly.

How we got here: The first draft computed avg(lat), avg(lon) per bucket. The review flagged the mathematical bias. Rewrote stage 1 to emit raw sum(lat), sum(lon), count(*) and divide in stage 2.

04Time-only output partitioning — never intersection_id⌄

Why: Partitioning Parquet output by intersection_id would recreate the skew at write time — one writer task per intersection, the hot one running forever. Partitioning by (region, event_date, event_hour) gives roughly 400 / (regions × hours) writer tasks per output directory, spreading the hot intersection across multiple writers automatically.

Rejected: Partition by intersection_id for query speed. Moves the failure from aggregation to write — same system, same hot key, different symptom. Spatial queries are kept fast via a Glue Catalog secondary index instead.

P4DevSecOps

Secure Weather API key rotation

They asked for

A Python Lambda for AWS Secrets Manager rotation of the Weather API key, with the explicit invariant that the new key is verified against the live API before the old key is deprecated.

What I built

A four-step rotation Lambda implementing the AWS Secrets Manager protocol modeled as a two-phase commit: the prepare phase (createSecret → setSecret → testSecret) verifies all participants; the commit phase (finishSecret) is the atomic state transition that moves AWSCURRENT. Idempotent at every step, body-redacted on error logs.

Why it works

Two-phase commit guarantees atomic key cutover — if any prepare step fails, the rotation aborts and the old key keeps serving traffic. The four-step contract is mandated by AWS Secrets Manager and provides the retry semantics; idempotency at each step makes Secrets Manager's automatic retries safe. Information-theoretically, redacting response bodies prevents the issued key (128+ bits of entropy) from being leaked to CloudWatch via debug echoes.

Pros · why this approach wins

  • 2PC atomicity — AWSCURRENT moves only after all preconditions verified; no partial-commit state.
  • Liveness preserved — old key continues to serve production traffic until the new key is verified; verification failure leaves the system in a healthy state.
  • Idempotent at every step — Secrets Manager retries are safe; create_secret short-circuits if AWSPENDING already exists; finish_secret short-circuits if the token is already AWSCURRENT.
  • Information-theoretic leak prevention — only resp.status and len(resp.data) are logged on error; the issued key's full entropy is preserved.
  • No bootstrap deadlock — admin bearer token lives in a separate Secrets Manager entry; rotating the Weather key never requires the Weather key. Breaks the classic "credentials to rotate credentials" cycle.
  • Race-condition resistance — catches both ResourceNotFoundException and InvalidRequestException on the createSecret probe, covering the TOCTOU window between Secrets Manager's version registration and label write.
  • Defensive admin-token parsing — accepts raw string and JSON {token: ...}; future-proof against admin-secret schema changes.
  • Provable invariant — smoke-tested in screenshots/run-rotation-smoke-test.py: AWSCURRENT remains on OLD through createSecret / setSecret / testSecret, only moves under finishSecret. A 401 in testSecret cleanly aborts and leaves the old key live.

Advanced techniques · references

  • Two-phase commit — Gray 1978, "Notes on Database Operating Systems"; Lampson & Sturgis 1976, "Crash Recovery in a Distributed Data Storage System." The canonical atomic-commit protocol; AWS Secrets Manager's four-step rotation maps directly onto 2PC's prepare/commit phases.
  • Idempotency in distributed systems — Pat Helland, "Idempotence Is Not a Medical Condition," ACM Queue 2012. The retry-safety pattern.
  • Saga pattern — Garcia-Molina & Salem, SIGMOD 1987, "Sagas." Long-running transaction model with compensating actions; the AWS rotation contract is a Saga with built-in compensation (old key keeps serving).
  • TOCTOU race conditions — Bishop & Dilger, "Checking for Race Conditions in File Accesses," CS 1996. The InvalidRequestException-vs-ResourceNotFoundException distinction is the AWS-specific manifestation.
  • Bootstrap trust problem — chicken-and-egg of needing credentials to rotate credentials. Solved by decoupling the admin token's rotation policy from the workload key's; PKI literature calls this "trust anchor separation."
  • Information leakage via logs — Schneier & Kelsey 1999, "Secure Audit Logs to Support Computer Forensics." The reverse problem: logging too much breaks the secret. len(resp.data) alone leaks negligible entropy.
  • AWS Secrets Manager rotation template — official AWS reference Lambda; defines the four-step contract this implementation conforms to.
1createSecretIssue new key, store AWSPENDING.
2setSecretNo traffic moves. Old key still serves.
3testSecretCall Weather API with new key.
4finishSecretPromote only after test passes.

Decisions & rationale

01testSecret runs before finishSecret moves AWSCURRENT⌄

Why: The assignment's explicit requirement. If AWSCURRENT were promoted first and only then tested, a broken new key would immediately take production traffic. Making finishSecret the only function that moves AWSCURRENT — and only after testSecret returns a 2xx from the live Weather API — means the old key keeps serving until we are sure the new one works.

How we got here: The invariant is smoke-tested in screenshots/run-rotation-smoke-test.py. Walking the four steps with mocked Secrets Manager + Weather API: AWSCURRENT stays on the OLD version through createSecret, setSecret, and testSecret, and only moves to NEW when finishSecret runs. A 401 from testSecret aborts cleanly and leaves AWSCURRENT on OLD.

02Separate admin-token secret (WEATHER_ADMIN_TOKEN_ARN)⌄

Why: This Lambda rotates the Weather API key. The admin bearer token used to issue new keys lives in a different Secrets Manager secret. If it lived in the same secret, rotation could deadlock — we'd need a working admin token to rotate the admin token. Keeping them separate breaks the cycle and lets each rotate on its own schedule.

Rejected: Hard-code the admin token in the Lambda — obvious security failure. Rotate both keys inline — couples failure modes and risks lockout.

03Redact response bodies in error logs (not just success logs)⌄

Why: The Weather admin API echoes the issued key back in error responses too — especially 4xx that include the key for debugging. Logging resp.data in any path would leak the new key to CloudWatch, defeating the whole rotation. The Lambda logs resp.status and len(resp.data) only.

How we got here: The first draft used f"…body={resp.data!r}" in the error message. The parallel-agent review pass flagged this as a critical security issue. All HTTP error logs now use a fixed format with the body redacted.

04Catch InvalidRequestException alongside ResourceNotFoundException⌄

Why: Secrets Manager retries each step. On retry, the AWSPENDING version may exist with the staging label but without a value yet — a brief race between Secrets Manager's registration of the new version and our write. That state raises InvalidRequestException, not ResourceNotFoundException. Catching both and proceeding to write is the correct idempotent behaviour.

Rejected: Catch only ResourceNotFoundException and let InvalidRequestException crash. Produces spurious failures and red CloudWatch alarms on perfectly normal retry paths. We also skip the staging-label check for createSecret specifically, matching the official AWS rotation template — the label write hasn't propagated yet at that step.
